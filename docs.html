Tower

Incremental Physics Builder

📦 13 modules 📝 ~6,000 lines ⚡ 60 fps target 🧱 3 materials 🏆 20 achievements

Overview

Tower is an incremental / idle physics builder built with vanilla HTML5 Canvas and Matter.js. Every block has real mass, every weld carries real stress, and every collapse is simulated in real time.

Core Loop

  1. Place blocks to grow your tower skyward.
  2. Earn passive income that scales with tower height.
  3. Buy upgrades to reduce stress, automate building, and unlock new materials.
  4. Prestige to reset the visible tower in exchange for permanent income multipliers.
  5. Unlock achievements as you hit milestones — then reach even higher.

Quick Start

No build step, no bundler, no install. Open in any modern browser:

bashcd tower
python3 -m http.server 8080
# or
npx serve .

Then open http://localhost:8080.

Note: Matter.js is loaded from a CDN (matter-js 0.20.0). An internet connection is required on first load.

Module Architecture

The game is organised as 13 ES6 modules coordinated by a strict GameLoop with a fixed per-frame execution order. Each module follows the same lifecycle: init(state), update(dt), serialize(), deserialize(data, state).

9RenderManagerCanvas 2D: sky, parallax, blocks, stress heatmap, floating text
10UIManagerDOM HUD: resources, upgrades, prestige panel, achievement toaster

Game Loop Execution Order

The GameLoop uses requestAnimationFrame with a hard cap of 50 ms per frame to prevent physics spiral-of-death after tab-away. The execution order within each tick is:

fixed order 1. Input           ← ghost calc, placement, camera
 2. Economy         ← passive income, offline catch-up
 3. Stats           ← detect maxHeight / prestige changes
 4. Achievements    ← check locked achievements
 5. Automation      ← AI block placement
 6. Physics         ← Matter.js step, constraint.reaction set
 7. Events          ← wind gusts, micro-impulses
 8. Stress          ← read constraint forces, break welds
 9. Render          ← Canvas drawing (sky → blocks → FX)
10. UI              ← DOM HUD (throttled to 100 ms)
11. Autosave        ← interval-based (every 30 s)

The critical ordering is Physics → Events → Stress: wind is applied AFTER the physics step but BEFORE the stress check, so wind loads are baked into the constraint reaction vectors when the solver evaluates them.

Single Source of Truth

All mutable game data lives in one serialisable GameState object. Every module receives the same reference at init and reads / writes through it. No module owns private copies of global state.

Physics Model

Matter.js handles all rigid-body dynamics with custom weld constraints that simulate structural joints.

Materials

IDNameDensityFrictionWeld StiffnessWeld Max Force
0Concrete0.00250.80.6580
1Steel0.0040.60.85200
2Reinforced0.0050.70.93400

Welds

When a block is placed, the PhysicsEngine finds any blocks directly below it (within 6 units vertically, with horizontal overlap) and creates a Matter.Constraint with the material's weldStiffness and weldMaxForce.

Stress & Failure

After every physics step, the StressSolver reads constraint.reaction — the instantaneous force vector the constraint is applying to keep the bodies together. If |reaction| > maxForce, the weld snaps (the constraint is removed from the world). The block becomes loose and falls.

Block stress (displayed in the UI) is the average constraint force across all welds on that block, normalised so 1.0 = at the breaking point.

Wind

Altitude-scaled horizontal force applied each frame during a wind event:

mathforce(y) = baseForce × (1 + |y| / heightScale)

This models the atmospheric boundary layer — wind speed increases with height because surface friction diminishes.

Collision Culling

Blocks that fall below y = 5000 are removed from both the Matter world and GameState to prevent orphan bodies from consuming CPU cycles.

Upgrade System

Upgrades are defined as data in js/data/upgrades.json and loaded synchronously by the EconomyManager.

Schema

json{
  "id": "reinforced_steel",
  "name": "Reinforced Steel",
  "tier": 1,
  "maxLevel": 50,
  "baseCost": { "money": 100, "steel": 50 },
  "costMultiplier": 1.15,
  "effectType": "stress_reduction",
  "baseValue": 0.03,
  "valuePerLevel": 0.03
}

Cost Formula

mathcost(resource) = baseCost[resource] × costMultiplier ^ level

Effect Query

Any module calls economy.getEffectValue('stress_reduction') which sums baseValue + level × valuePerLevel across all upgrades of that type.

Prestige Cycle

Once maxHeight ≥ 500, the prestige button in the side panel becomes active.

Reward Formula

mathbase     = ⌊√(maxHeight × 0.5)⌋
amplifier = 1 + getEffectValue('prestige_amplifier')
tokens    = ⌊base × amplifier⌋

Examples:
  height   500 →  15 tokens
  height  2000 →  31 tokens
  height  8000 →  63 tokens

Income Multiplier

Each held prestige token adds +10 % to money and steel income:

mathmultiplier = 1 + tokens × 0.10

Reset Preserves

Prestige tokens, multipliers, upgrade levels, settings, collapse count, and play time survive the reset. Tower blocks, resources, and camera position are wiped to a fresh state.

Audio Engine

All sounds are procedurally generated via the Web Audio API — no audio files, no CORS issues, no preloading.

Block placed80–120 Hz sine, 120 ms decay
UI click800–1200 Hz sine, 40 ms decay
CollapseSawtooth sweep 60→20 Hz + white noise burst, 500 ms
AchievementC5→E5→G5 ascending arpeggio
AmbienceA1 (55 Hz) + A0 (27.5 Hz) sine drones, looped

The AudioContext is created on the first user gesture (click / keydown) per browser autoplay policy. Volume is controlled by GameState.settings with separate master, SFX, and music gain nodes.

Rendering Pipeline

The RenderManager draws in this order each frame (after the camera transform is applied):

  1. Sky gradient — deep-space black to horizon blue, shifts with camera Y
  2. Mountains — two parallax layers (depth 0.5 / 0.7), jagged silhouettes
  3. Clouds — 30 ellipses at parallax depth 0.15–0.35
  4. Ground line — subtle glow + grid
  5. Blocks — coloured by material, frustum-culled, 3-px bevel, stress red-tint overlay
  6. Stress heatmap — constraint lines coloured green→yellow→red based on |reaction| / maxForce
  7. Snap particles — white circles that grow and fade when a weld breaks
  8. Ghost preview — translucent green (valid) or red (invalid) at cursor
  9. Floating text — "+$15", "-10" popups that float upward and fade out
  10. Crosshair — subtle centre-screen indicator
Performance: Blocks outside the view frustum (plus 100 px margin) are culled. Stress lines are culled if both endpoints are off-screen. Floating text is trivially cheap.

Save System

Saves are written to localStorage under the key tower_save.

  • Autosave every 30 seconds
  • On-unload save when the page closes
  • Manual wipe button (double-click confirmation) in the bottom-right corner

The save payload includes:

json{
  "schema":     1,
  "savedAt":    1712345678000,
  "playTime":   3600,
  "maxHeight":  1200,
  "gameState":  { /* full serialisable state */ },
  "modules": {
    "PhysicsEngine":    { "bodies": [...], "constraints": [...] },
    "AutomationManager": { "nextId": 42, "cooldown": 1.2 },
    ...
  }
}
Validation: Each load goes through a 4-step pipeline: JSON parse → schema version check → isValidState() structural validation → field backfill for older saves. Corrupt data is silently discarded.

Audit & Known Issues

The codebase was subjected to a full systems audit covering state integrity, physics memory management, math safety, and rendering performance. 12 issues were identified and fixed.

Critical Fixes Applied

#IssueFix
1EventManager.js: const Matter = window.Matter; was inside a JSDoc /** */ block — micro-impulse events crashed with ReferenceErrorMoved the declaration outside the comment block
2main.js: StressSolver patch removed blocks from GameState but never called physics.destroyBlock() — orphan Matter bodies leaked on every collapseAdded physics.destroyBlock(blockId) before GameState removal
3PhysicsEngine.deserialize(): Created a new ground body each save-load without removing the old one — accumulating duplicate groundsAdded Composite.remove() of old ground before rebuilding
4main.js + GameLoop.js: SaveManager was never registered in the phase list — autosave timer searched for a phase named 'SaveManager' that didn't existAdded loop.register({name:'SaveManager', ...}) to the phase list

High / Medium Fixes

#IssueFix
5main.js: input._placeBlock was completely replaced (duplicating all placement logic) instead of wrapping the originalChanged to call origPlaceBlock() then inject hooks
6main.js: Used ui._short() — a private UIManager method, breaking encapsulationMoved formatNumber/formatShort to math.js as exported functions, imported by both UIManager and main.js
7main.js: Same replacement pattern for automation._autoPlaceBlockChanged to wrapper pattern
8EconomyManager.js: Height factor was additive in income formula (income started at 2× at height 0)Changed to multiplicative: base × prestige × heightFactor × (1 + upgradeBonus)
9PhysicsEngine._cullFallenBlocks(): Incremented totalCollapses for harmless debris removalRemoved the increment (collapses are tracked by StressSolver only)
10UIManager._eng(n): Returned "Infinityundefined" for NaN inputsDelegated to formatNumber() from math.js which has a !Number.isFinite guard
11math.js costExponential(): Overflows to Infinity at extreme upgrade levelsAdded !Number.isFinite(result) guard returning MAX_SAFE_INTEGER
12AchievementManager.js: "Self-Aware" referenced state.stats.otherAchievements which doesn't existAdded other_achievements condition type counting unlocked achievements
Architecture Invariants Preserved: Single source of truth (shared GameState), strict 11-phase GameLoop ordering, deterministic pure-function math, safe persistence with silent corruption handling, and decoupled cross-module communication via GameState mutation + the StatTracker event bus.

Development Guide

Adding an Upgrade

  1. Add its definition to js/data/upgrades.json AND the UPGRADE_DEFS array in EconomyManager.js.
  2. Use economy.getEffectValue('your_effect_type') in the consuming module.

Adding an Achievement

  1. Add its definition to js/data/achievements.json AND the ACHIEVEMENT_DEFS array in AchievementManager.js.
  2. Available condition types: stat_ge, max_height_ge, prestige_count, upgrade_level.
  3. Stats emitted by StatTracker: blocks_placed, money_earned, steel_earned, research_earned, collapse, upgrade_purchased, prestige, max_height.

Module Conventions

Every module follows the same interface:

  • constructor(...dependencies) — receives external refs
  • init(state) — receives shared GameState, builds internal structures
  • update(dt) — per-frame logic (if registered in GameLoop)
  • serialize() — returns module-specific data or null
  • deserialize(data, state) — restores from serialized data
